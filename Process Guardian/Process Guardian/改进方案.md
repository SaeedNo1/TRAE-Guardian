# Process Guardian 改进方案

> 目标：从"杀毒软件"升级为终端安全平台（EDR+AV），架构设计更专业、可扩展。

---

## 总体架构

```
                GUI（Qt）
                    │
      ┌─────────────┴─────────────┐
      │                           │
 配置中心                     日志中心
      │                           │
      └─────────────┬─────────────┘
                    │
            AI风险评分中心
                    │
 ┌──────────┬──────────┬──────────┐
 │          │          │          │
静态扫描   行为监控   底层监控   响应中心
```

## 一、GUI（Qt）

仅负责显示，不负责检测。

包括：
- 主页
- 实时风险
- 进程列表
- 事件日志
- 病毒隔离区
- 启动项管理
- 服务管理
- 计划任务
- 网络连接
- 磁盘状态
- MBR状态
- AI分析结果
- 设置
- 更新

以后所有数据都来自后台线程。

---

## 二、后台线程

建议拆成以下模块：

### ① ETW线程

负责：
- Process
- Thread
- ImageLoad
- Registry
- Network
- DNS
- FileIO
- Kernel
- Service
- Driver

作用：全部送给AI评分。

### ② 新进程监控线程

ETW发现 CreateProcess 后：
- 立即加入观察名单
- 持续20ms扫描句柄，持续5秒
- 然后移除观察名单

### ③ 全局句柄扫描线程

- 1000ms遍历全部进程
- 检查：PhysicalDrive、Volume、RawDisk、驱动设备、系统对象
- 发现危险时立即进入高速扫描模式

### ④ 高速扫描线程

只针对危险进程：
- 10ms扫描句柄
- 持续10秒

### ⑤ MBR线程

- 开机时备份MBR
- 低频（5秒）读取MBR
- 如果发现PhysicalDrive打开，立即50ms MBR Hash
- 恢复后继续低频

### ⑥ 注册表线程

维护：
- Run / RunOnce
- Services
- Explorer
- Shell
- Image File Execution
- Browser

支持：快照、恢复、积分

### ⑦ 文件线程

负责：
- SHA256
- PE解析
- 数字签名
- YARA
- 字符串
- 导入表/导出表
- 熵
- 壳
- 版本信息

### ⑧ 网络线程

负责：
- TCP/UDP
- DNS
- HTTP/TLS
- 下载/上传

记录：IP、域名、国家、进程

### ⑨ AI线程

输入：ETW、句柄、注册表、网络、PE、磁盘

输出：风险解释、建议、加分、减分、未知样本分析

**AI不要直接删东西。**

## 三、风险评分中心

建议每个模块独立积分：

| 模块 | 分数 |
|---|---|
| 静态 | 25 |
| 行为 | 40 |
| 底层 | 18 |
| AI | +8 |
| **总计** | **91** |

达到80分 → 处置

这样方便调试。

---

## 四、响应中心

不要只有 Terminate 一种响应。建议分等级：

- 限制网络
- 暂停线程
- 挂起
- 终止
- 隔离
- 删除
- 恢复
- 黑名单

---

## 五、白名单

不要只有文件名。建议综合：

- SHA256
- Publisher
- 签名
- 路径
- 版本
- 用户确认

## 六、日志数据库

建议使用 SQLite，保存：
- 所有ETW
- 风险
- AI
- 恢复
- 删除
- 网络
- 扫描历史

以后GUI查询即可。

## 七、配置中心

- 扫描频率
- 白名单
- AI
- 日志
- 更新

## 额外建议

### 1. 引入状态机管理风险

不要让每个模块直接决定处置，而是让进程在几个状态间转换：

```
Unknown
    ↓
Observed（新进程，重点观察）
    ↓
Suspicious（达到一定积分）
    ↓
Restricted（限制网络、禁止启动项）
    ↓
Quarantined（隔离）
    ↓
Removed（清除）
```

所有模块只负责提供证据，响应中心根据状态统一处理，逻辑更清晰。

### 2. 规则引擎

不要把所有积分写成 if...else。建议把规则写成可配置的形式：

| 字段 | 示例 |
|---|---|
| 规则ID | R001 |
| 条件 | 打开 PhysicalDrive + 尝试写入 |
| 积分 | +35 |
| 严重等级 | High |
| 说明 | 疑似 MBR 篡改 |

以后新增规则、调整分值甚至在线更新规则，都不需要重新编译程序。
