# TRAE Guardian 功能介绍与核心架构

## 一、项目简介

TRAE Guardian 是一款**纯用户态轻量化EDR+AV一体化终端安全防护平台**，基于Qt构建可视化管理界面，采用分层解耦多线程架构，搭载统一AI风险评分中心、可配置规则引擎、进程状态机管控、标准化插件拓展接口，集成静态病毒扫描、全维度行为监控、底层磁盘/引导防护、分级自动响应、自保护防篡改、本地日志数据库六大核心体系，仅73MB体积实现个人终端、小型办公终端全场景安全防护。

## 二、核心功能

### 2.1 进程管理与监控

- **实时进程列表**：展示系统所有进程及其详细信息（PID、内存占用、签名状态等）
- **进程终止**：支持单进程终止和树杀（终止进程及其子进程）
- **重复查杀**：自动监控并重复杀死指定进程，防止看门狗复活
- **新进程观测**：ETW捕获新进程创建后，自动进入5秒高频监控

### 2.2 恶意行为检测

- **物理磁盘访问拦截**：通过句柄扫描检测`PhysicalDrive`、`Volume`等敏感设备访问
- **MBR防护**：开机备份MBR哈希，实时校验完整性，检测到篡改自动恢复
- **注册表监控**：监控Run键、服务配置、浏览器劫持项等高危路径
- **文件静态扫描**：PE结构解析、数字签名校验、SHA256哈希、恶意API导入检测

### 2.3 网络审计

- **TCP/UDP连接记录**：记录所有进程网络连接
- **DNS解析审计**：监控域名解析行为
- **网络访问限制**：可限制可疑进程网络访问

### 2.4 AI智能分析

- **多维度风险评分**：静态(25分) + 行为(40分) + 底层(18分) + AI(±8分)
- **规则引擎**：可配置规则，支持通配符匹配，无需修改源码即可新增规则
- **分级响应**：8级精细化处置策略（观测→通知→限制网络→暂停→终止→隔离→删除→黑名单）

### 2.5 自保护机制

- **GUI/守护进程互保**：一方被终止，另一方自动拉起
- **文件保护**：程序文件被删除/覆盖时自动恢复
- **配置保护**：防护配置被篡改时一键回滚

### 2.6 可视化界面

- **实时风险看板**：展示系统安全状态
- **事件日志**：完整安全事件记录，支持检索和导出
- **进程列表**：全量进程观测，支持筛选和操作
- **AI对话**：基于SiliconFlow的自然语言交互

### 3.2 核心模块详解

#### 3.2.1 事件驱动架构

系统采用统一事件分发器`DispatchDetectEvent`串联所有模块：

```c
// daemon_core.c - 事件分发器
void DispatchDetectEvent(DaemonState* ds, DetectEvent* event) {
    // 1. 规则引擎匹配
    RuleMatchResult result = RuleEngineMatch(ds->rules, event);
    
    // 2. 评分中心打分
    ScoreEntry* score = ScoreCenterUpdate(ds->scores, event, result.score);
    
    // 3. 状态机状态转换
    StateMachineTransition(ds->sm, event->pid, score->totalScore);
    
    // 4. 响应中心执行动作
    ResponseCenterExecute(ds->rc, event, score->totalScore);
}
```

**支持的事件类型**：
- `DETECT_EVENT_PROCESS_SPAWN`：进程创建
- `DETECT_EVENT_REGISTRY_WRITE`：注册表写入
- `DETECT_EVENT_REGISTRY_DELETE`：注册表删除
- `DETECT_EVENT_DISK_ACCESS`：磁盘访问
- `DETECT_EVENT_FILE_MODIFY`：文件修改
- `DETECT_EVENT_NETWORK_CONNECT`：网络连接

#### 3.2.2 规则引擎

规则引擎支持多条件组合匹配和通配符模式：

```c
// rule_engine.c - 规则定义
typedef struct {
    wchar_t id[16];           // 规则ID，如 R001
    wchar_t name[128];        // 规则名称
    wchar_t description[512]; // 规则描述
    RuleCondition conditions[MAX_CONDITIONS];
    int score;                // 匹配成功后增加的分数
    ResponseAction action;    // 默认响应动作
} DetectionRule;

// 示例规则：未签名进程访问物理磁盘
// R001: Unsigned process accessing physical disk
// conditions: [UNSIGNED, PATH: *PhysicalDrive*]
// score: +35, action: TERMINATE
```

#### 3.2.3 评分中心

多维度加权评分体系：

```c
// score_center.c - 评分结构
typedef struct {
    DWORD pid;
    wchar_t name[64];
    int staticScore;        // 静态扫描: 0-25分
    int behaviorScore;      // 行为监控: 0-40分  
    int kernelScore;        // 底层监控: 0-18分
    int aiScore;            // AI研判: -10~+8分
    int totalScore;         // 总分 = 静态+行为+底层+AI
} ScoreEntry;

// 分级阈值
#define THRESHOLD_OBSERVED    10   // 进入观测状态
#define THRESHOLD_SUSPICIOUS  35   // 标记可疑
#define THRESHOLD_RESTRICTED  50   // 限制权限
#define THRESHOLD_QUARANTINED 70   // 隔离处置
#define THRESHOLD_REMOVED     80   // 彻底清除
```

#### 3.2.4 进程状态机

6态单向流转状态机，确保处置逻辑清晰：

```c
// state_machine.c - 状态定义
typedef enum {
    PROC_STATE_UNKNOWN = 0,        // 未知
    PROC_STATE_OBSERVED,           // 观测中
    PROC_STATE_SUSPICIOUS,         // 可疑
    PROC_STATE_RESTRICTED,         // 受限
    PROC_STATE_QUARANTINED,        // 隔离
    PROC_STATE_REMOVED,            // 清除
    PROC_STATE_WHITELISTED,        // 白名单
    PROC_STATE_TRUSTED             // 信任
} ProcessState;

// 状态流转规则：只能升级，不能降级（白名单/信任除外）
// Unknown → Observed → Suspicious → Restricted → Quarantined → Removed
```

#### 3.2.5 响应中心

8级分级响应执行引擎：

```c
// response_center.c - 响应动作
typedef enum {
    RESPONSE_NONE = 0,
    RESPONSE_OBSERVE,              // 观测
    RESPONSE_NOTIFY,               // 通知
    RESPONSE_LIMIT_NETWORK,        // 限制网络
    RESPONSE_SUSPEND_PROCESS,      // 暂停进程
    RESPONSE_TERMINATE,            // 终止进程
    RESPONSE_QUARANTINE,           // 隔离文件
    RESPONSE_DELETE,               // 删除文件
    RESPONSE_BLACKLIST             // 加入黑名单
} ResponseAction;
```

### 3.3 线程模块

#### 3.3.1 ETW监控线程

捕获进程创建、线程注入、模块加载、注册表修改、网络连接等内核事件。

#### 3.3.2 句柄扫描线程

使用`NtQuerySystemInformation`枚举所有进程句柄，检测敏感设备访问：
- **低频模式**：1000ms间隔，全局扫描
- **高频模式**：20ms间隔，针对新进程和可疑进程

#### 3.3.3 MBR防护线程

- 开机备份512字节MBR哈希
- 常态5秒低频校验
- 检测到物理磁盘访问时切换50ms高频校验
- 哈希不一致时自动恢复

#### 3.3.4 注册表监控线程

使用`RegNotifyChangeKeyValue`监控Run键等高危路径，检测到修改时触发事件分发。

## 四、关键技术亮点

| 技术创新点 | 实现方式 | 解决的问题 |
|-----------|---------|-----------|
| **实时句柄扫描** | `NtQuerySystemInformation`枚举所有进程句柄 | 前置拦截物理磁盘访问，杜绝MBR篡改 |
| **MBR哈希校验** | 开机备份512字节MBR，双频校验 | 检测MBR篡改并自动恢复 |
| **进程状态机** | 6态单向流转，各模块仅提交证据 | 消除多模块处置冲突，逻辑清晰 |
| **多维度评分** | 静态(25)+行为(40)+底层(18)+AI(±8)加权计分 | 精准风险评估，减少误报漏报 |
| **可配置规则引擎** | 规则以JSON存储，支持通配符匹配 | 新增规则无需修改源码 |
| **分级响应** | 8级精细化处置策略 | 避免过度处置，保护正常程序 |
| **事件驱动架构** | 统一事件分发器串联所有模块 | 模块解耦，易于拓展 |
| **自保护机制** | GUI/守护进程互保，文件自动恢复 | 防止恶意程序破坏防护体系 |
