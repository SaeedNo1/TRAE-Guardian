2. 技术分析

2.1 极简导入表 — 刻意隐藏行为

所有 4 个样本的导入表仅包含 KERNEL32.dll，这是 Play 最核心的反分析技术之一：

可疑 API用途风险等级
LoadLibrary + GetProcAddress运行时动态解析所有其他 API（加密、网络、文件操作），静态分析无法看到真实行为极高
IsDebuggerPresent检测调试器附加，触发反调试路径高
TerminateProcess终止安全软件进程或自毁极高
FreeLibrary卸载已注入的 DLL高
send 仅 Sample 04网络数据外传 — 双重勒索的数据窃取阶段极高

这意味着：传统静态扫描只能看到 5-6 个无害的 KERNEL32 API，真正的恶意行为（文件加密、网络传播、防御禁用）全部通过 GetProcAddress 动态获取，杀毒软件的静态特征库根本扫不到。

2.2 RSA 加密引擎

所有样本均包含 RSA1 字符串标记和完整的数学函数库：

// 从 strings.txt 中提取的数学函数 log10, asin, acos, sqrt, sinh, cosh, tanh, atan, atan2, ceil, floor, fabs, modf, ldexp, _cabs, _hypot, fmod, frexp, _logb, _nextafter // RSA 公钥标记 RSA1 // RSA-4096 公钥头 // 加密方案: AES-RSA 混合加密 // AES-256 对称加密文件内容 // RSA-4096 加密 AES 密钥 // 间歇加密: 每 0x100000 字节只加密一半

Play 使用 AES-256 + RSA-4096 混合加密，且采用间歇加密（intermittent encryption）——每 1MB 数据只加密其中一半，这样加密速度快 2 倍，同时安全监控软件检测到文件被大幅修改的概率也更低。

2.3 多层反分析技术

Play 勒索病毒的反分析技术在勒索病毒界堪称顶级，以下是已确认的技术：

1. ROP 控制流混淆

使用 100+ 个函数修改栈顶返回地址，使 IDA/Ghidra 反编译器无法正确追踪执行流。每个 call 指令后跟垃圾字节，真实代码跳转到其他位置。

2. SEH Hook 异常处理劫持

手动修改线程异常注册链表，触发异常后将执行流转移到隐藏的密钥生成函数。密钥不存储在文件中，而是运行时动态计算。

3. 反沙箱延时

密钥生成函数包含一个 9 亿次迭代的循环，耗时约 30-35 秒。沙箱通常在此期间超时，不会记录任何恶意行为。

4. 动态 API 解析

导入表只有 KERNEL32.dll，所有其他 API（加密、网络、文件）通过 hash 查找在运行时动态获取。

5. 反调试检测

IsDebuggerPresent 检查调试器，可能还有硬件断点检测、时序检测等。检测到调试器后走无害路径。

6. 每次部署重新编译

每次攻击使用重新编译的二进制文件，hash 值不同，传统特征码检测无效。

3. 为什么杀不掉它

Defender 失败的原因 Play 勒索病毒使用多层手段对抗 Windows Defender，这就是为什么你的软件和 Defender 都报毒但无法清除的原因：
对抗手段技术细节效果
GMER / PowerTool 内核级杀软禁用 Play 操作者使用 GMER 和 PowerTool 等内核级工具，直接修改内核结构来禁用杀毒软件的回调函数和驱动保护 Defender 的实时保护被从内核层关闭，扫描引擎虽运行但无法拦截文件操作
PowerShell 脚本禁用 Defender 通过 PowerShell 修改注册表 Set-MpPreference -DisableRealtimeMonitoring $true，禁用 Tamper Protection Defender 实时监控被关闭，无法检测后续加密行为
IOBit 禁用杀软 利用合法的 IOBit 软件作为代理来禁用安全软件 绕过行为检测，因为 IOBit 是合法签名软件
删除卷影副本 使用 ALPHAVSS 工具删除所有 Volume Shadow Copies (vssadmin delete shadows /all) 系统恢复点被清除，无法通过系统还原恢复文件
清除事件日志 使用 wevtutil cl 清除所有 Windows 事件日志 取证痕迹被清除，难以追踪入侵路径
进程注入 注入到合法进程 (explorer.exe, svchost.exe) 中运行 即使检测到恶意行为，杀软也无法终止合法系统进程
每次重新编译 每次部署使用重新编译的二进制文件，hash 不同 基于 hash 的特征检测完全失效

4. 蓝屏 (BSOD) 原因分析

蓝屏的直接原因 你描述的"重启卡死 → 强制断电 → 蓝屏"现象，是 Play 勒索病毒多个行为叠加的后果：
阶段 1: 低调潜伏 (你说"只有一个进程")

病毒进入系统后首先进行网络侦察：扫描内网、收集凭据、枚举 Active Directory。这个阶段只有一个进程运行，行为低调，不触发加密。病毒使用 AdFind、Bloodhound 等工具收集网络拓扑信息。

阶段 2: 禁用防御

病毒使用 GMER / PowerTool 从内核层禁用 Defender 的回调函数。这些工具直接修改内核内存结构（如 PsSetCreateProcessNotifyRoutineEx 回调），如果此时系统尝试正常重启，内核状态不一致 → 重启卡死。

阶段 3: 强制断电 → 蓝屏

你强制断电后，内核中的非法修改没有被正确清理。开机时：

  • 被篡改的内核回调函数指向无效内存 → PAGE_FAULT_IN_NONPAGED_AREA
  • Defender 驱动初始化时发现回调被移除 → DRIVER_VERIFIER_DETECTED_VIOLATION
  • 卷影副本服务 (VSS) 被删除后系统恢复尝试访问无效引用 → SYSTEM_SERVICE_EXCEPTION
  • Play 可能还修改了 BCD (Boot Configuration Data) 或注册表 Run 键 → 启动循环崩溃
阶段 4: 已知崩溃 Bug

根据 CyberArk 分析，Play 本身存在一个多线程崩溃 Bug：网络扫描的 ping 线程在 1.4 秒超时后释放内存，但其他线程可能仍在访问该内存 → 访问违例 → WERFault 触发。这个 Bug 在某些情况下也会导致进程崩溃并留下不完整的内核修改。

7. 针对你的系统的建议

⚠ 紧急措施 你的电脑已经蓝屏过，说明病毒可能已经修改了系统底层结构。以下是建议操作顺序：
  1. 不要在原系统上尝试恢复 — 内核可能已被篡改，任何在原系统上的操作都不可信
  2. 使用 Windows PE / Linux Live USB 启动 — 从外部介质引导，完全绕过被篡改的 Windows 内核
  3. 检查以下关键位置：
    • C:\Users\Public\Music\ — Play 的工具和勒索信存放目录
    • C:\ProgramData\SkyPDF\ — CLFS 漏洞利用痕迹 (PDUDrv.blf, clssrv.inf)
    • 注册表 HKLM\SOFTWARE\Microsoft\Windows Defender — 检查 Defender 是否被禁用
    • 注册表 HKLM\SYSTEM\CurrentControlSet\Services — 检查可疑服务
    • 计划任务 — Play 会创建计划任务维持持久化
  4. 检查文件是否已被加密 — 搜索 .PLAY 扩展名的文件，或 ReadMe.txt 勒索信
  5. 重装系统是最安全的选择 — 鉴于内核已被篡改且 Defender 被禁用，建议全盘格式化重装
  6. 重装后立即更新所有补丁，特别是：
    • CVE-2025-29824 (CLFS 驱动提权 — Play 在 2025 年利用的零日漏洞)
    • Exchange Server 补丁 (ProxyNotShell)
    • SimpleHelp RMM 补丁 (如果使用了该工具)
  7. 启用 Tamper Protection — 防止 PowerShell 修改 Defender 设置
  8. 开启 BitLocker — 全盘加密防止离线文件窃取

8. 对 Process Guardian 的启示

基于 Play 勒索病毒的分析，以下特征可以作为 Process Guardian 的检测规则：

行为特征检测

  • 极简导入表检测：32位 PE 文件仅导入 KERNEL32.dll 且包含 LoadLibrary + GetProcAddress → 高风险
  • 反调试行为：进程调用 IsDebuggerPresent 后改变执行路径
  • Defender 禁用检测：监控注册表 HKLM\SOFTWARE\Microsoft\Windows Defender\Real-Time Protection 的修改
  • 卷影删除检测：监控 vssadmin.exe、wbadmin.exe 的命令行参数
  • GMER/PowerTool 加载：这两个工具不应在正常环境中出现，一旦加载立即告警
  • 批量文件修改：短时间内大量文件被修改且写入大小约为 0x100000 (1MB) 的倍数 → 间歇加密特征
  • 事件日志清除：监控 wevtutil.exe cl 命令
  • C:\Users\Public\Music\ 写入：Play 的特征性目录

网络特征检测

  • 大量 SMB 连接尝试 (密码爆破)
  • 内网 ICMP 扫描 (Play 的 ping 线程行为)
  • 非预期的 WinSCP / WinRAR 进程启动
关于文件恢复 Play 使用间歇加密（每 1MB 只加密一半），这意味着未加密的部分可能可以恢复部分文件内容。可以使用 White Phoenix (CyberArk 开源工具) 尝试从被间歇加密的大文件中恢复部分内容。但前提是文件没有被完全覆盖。
